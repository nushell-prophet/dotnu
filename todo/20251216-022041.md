# `dotnu expand-code` — generate code from `#**` directive comments (the opposite of `dotnu embeds`)

## Idea

`dotnu embeds` runs commands and writes their **output** back into the file as `# =>` comments. I want the opposite: a command that runs a small pipeline written **inside a comment** and writes that pipeline's result back as **real code lines**, right after the comment.

The directive comment is never modified, so the generation can be re-run any time to refresh the code below it.

- Open marker: `#**`. Everything after it on the line is a Nushell pipeline.
- The pipeline must return text. Each line of that text becomes one generated code line, inserted right after the directive.
- End marker: `#**end` on its own line closes the generated block.
- Re-running replaces everything between `#**` and `#**end`; the directive and the end marker stay.

## Basic example

Source file:
```nushell
#** ls todo/ | get name | each { $"open -r ($in)" } | to text
#**end
```

After running the command:
```nushell
#** ls todo/ | get name | each { $"open -r ($in)" } | to text
open -r todo/20251216-022041.md
open -r todo/20260630-090000.md
#**end
```

Now the whole file is executable, and re-running the command refreshes the `open -r ...` lines whenever `todo/` changes.

## Why: integration tests triggered this

This repo's integration tests are the case that pushed me to write the idea down. Each test is a snapshot: run a command, save its output, compare against a fixture (`toolkit.nu`, `run-snapshot-test`). The list of tests is written by hand, so it can drift from the real fixtures, and a failure points at the whole block rather than one line.

A `#**` directive turns that list into generated code driven by the fixtures that actually exist on disk:
```nushell
#** ls tests/output-yaml/*.yaml | get name | path parse | get stem | each { $"    (run-snapshot-test '($in)' ...)" } | to text
```
The generated lines stay explicit — one call per line — so a failing test traces to a line number, and the list can never fall out of sync with the fixtures.

(The exact tests were reworked since I first had this idea; the point is the pattern, not these specific calls.)

## Decisions

- **Name: `dotnu expand-code`.** Not `expand` — that would clash with the builtin `expand`.
- **Boundary: an end marker.** `#**end` closes the block, so a re-run replaces only the lines between the directive and the end marker.

## Decision: separate parsing layer

`dotnu embeds` already scans a `.nu` file line by line to find its own markers (the `# =>` result comments), and `numd` scans markdown to find fenced code blocks. Both walk a file and locate special regions. `expand-code` needs the same kind of scan — find each `#**` and its matching `#**end`.

`expand-code` gets its own scanner. The marker shapes differ enough (`# =>` vs fenced blocks vs `#**`…`#**end`) that a shared helper would be mostly branching. Factor one out later only if the scanning code clearly repeats.

## Implemented

Commit `feat: add dotnu expand-code`. Three commands in `dotnu/commands.nu`: `expand-code` (public, in `mod.nu`), `find-expand-directives` (the line scanner), `run-expand-pipeline` (runs one directive's pipeline with `--no-config-file`, in the file's directory so relative paths resolve against it). Tests in `tests/test_commands.nu`; README section between Embeds and Dependency Analysis.

Decisions taken while implementing:

- **Fail-fast scanner.** Unclosed directive, stray `#**end`, or an empty pipeline each error with a line number, rather than silently no-op. The block must already contain both `#**` and `#**end` — the command fills between them, it does not insert a missing end marker.
- **No auto-indent.** Generated lines are written verbatim; the pipeline controls its own leading spaces (as the integration-test example does with `$"    (run-snapshot-test…"`). An indented `#**` directive does *not* auto-indent its output. ??? Revisit if a directive inside an indented block turns out to be a real use case — small addition.
- **Known limit: generated output must not itself contain `#**` / `#**end` lines.** The scanner reads the source line by line, so a pipeline that emits a `#**end` line would produce a file that mis-scans on the next run. This fails loudly (a scanner error with a line number), not silently — consistent with fail-fast. Pathological in practice: real generated code is not dotnu directives.
